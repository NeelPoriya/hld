> Category: Object Storage · Provider: AWS · License: Proprietary (managed)

## TL;DR
Amazon S3 (Simple Storage Service) is **the** reference cloud object store. It exposes a flat namespace of **buckets** and **keys** through HTTP `PUT/GET/DELETE`, with **eleven 9s of durability** (10⁻¹¹ annual object loss) and effectively unlimited capacity. It's the storage layer underneath the bulk of modern data infrastructure on AWS — data lakes (Iceberg / Delta / Hudi sit on top), backups, ML feature stores, static websites, and warehouse cold tiers (Snowflake, Redshift Spectrum, Athena).

## What problem does it solve?
**Block storage** (EBS) and **filesystems** don't scale to petabytes for many concurrent writers and don't span regions easily. S3 solves:
- **Unlimited capacity** — petabyte+ buckets are routine.
- **Durability** without operating storage hardware.
- **Concurrent multi-tenant access** with IAM-driven authorization.
- **Pay-per-byte storage + tiered classes** for hot/warm/cold/archive economics.
- **HTTP API** that any language / tool / framework can consume.

## When to use
- **Data lakes** — Parquet / Iceberg / Delta / Hudi tables.
- **Backups & archives** — point-in-time, immutable with Object Lock + Glacier.
- **Static website hosting** + **CloudFront** origin for global delivery.
- **ML training data** — large datasets read by EMR / SageMaker / EKS jobs.
- **Application object storage** — user uploads, exports, large blobs.
- **Cross-region disaster recovery** via Cross-Region Replication.

## When NOT to use
- **Low-latency random reads** of small objects — millisecond GETs are achievable but EBS / Redis is faster.
- **Filesystem semantics** (rename, append-in-place) — S3 is object-storage; use EFS or FSx instead.
- **OLTP** — wrong tool.
- **Strong-read-after-overwrite consistency on globally cached data** — read-after-write is now strongly consistent (since 2020), but downstream caches (CloudFront) may serve stale.
- **Small files at extreme rates** — 5,500 GET / 3,500 PUT per second per prefix; design key prefixes carefully.

## Data Model
- **Bucket** — globally unique name, region-bound, IAM-policy-bound.
- **Object** — key (string) + value (up to 5 TB) + metadata (system + user-defined).
- **Versioning** — optional per bucket; every overwrite/delete creates a new version.
- **Storage classes** — Standard, Intelligent-Tiering, Standard-IA, One Zone-IA, Glacier Instant / Flexible / Deep Archive.
- **Object Lock** — WORM (write-once-read-many) for compliance.
- **Multipart upload** — split big objects into 5 MB+ parts uploaded in parallel.

```python
import boto3
s3 = boto3.client("s3")

# Upload
s3.put_object(Bucket="my-bucket", Key="logs/2026/04/03.json", Body=b"...")

# Download
obj = s3.get_object(Bucket="my-bucket", Key="logs/2026/04/03.json")
data = obj["Body"].read()

# Multipart for large files
from boto3.s3.transfer import TransferConfig
config = TransferConfig(multipart_threshold=64 * 1024 * 1024, max_concurrency=10)
s3.upload_file("big.parquet", "my-bucket", "data/big.parquet", Config=config)

# Pre-signed URL (time-limited, no IAM creds needed by client)
url = s3.generate_presigned_url("get_object",
    Params={"Bucket": "my-bucket", "Key": "exports/report.csv"}, ExpiresIn=3600)
```

## Architecture & Internals (publicly known)
- **Distributed across multiple AZs** within a region (Standard class) — data redundantly stored across at least 3 AZs.
- **Erasure coding** internally; not simple replication, hence the "11 nines" durability.
- **Front-end** — load balancers route to a fleet of stateless request servers.
- **Indexing** — distributed key-value index (formerly known to use a horizontally partitioned index service); explains the "key prefix" hot-spotting concern (now mitigated since 2018 — prefixes auto-shard).
- **Strongly consistent** read-after-write since Dec 2020 (was eventual before).

## Consistency Model
- **Read-after-write** strongly consistent for new objects (PUT then GET → see it).
- **Read-after-overwrite** strongly consistent (since 2020).
- **List-after-write** strongly consistent.
- **No multi-object transactions** — each object is its own atomic unit.

## Replication
- **Multi-AZ within region** — built in.
- **Cross-Region Replication (CRR)** — async, opt-in per bucket.
- **Same-Region Replication (SRR)** — for compliance / log aggregation.
- **Replication Time Control (RTC)** — 99.99% replicated within 15 minutes SLA.
- **Multi-Region Access Points (MRAP)** — single endpoint, routes to nearest replica.

## Partitioning / Sharding
- **Key prefix sharding (auto, since 2018)** — S3 auto-shards based on key prefix patterns; no longer need random prefixes for hot prefixes (mostly).
- **Per-prefix limits** — 5,500 GET / 3,500 PUT/POST/DELETE per second; hit them with concurrent requests across many prefixes.
- **Hot-key pitfall:** millions of writes to keys sharing one prefix path → throttling. Use multi-segment prefixes (e.g., date or hash buckets) and parallelize across them.

## Scale
- Public references describe **trillions of objects** stored, **millions of requests per second** at peak per region.
- **Largest known buckets:** exabyte+.
- **Single object:** up to 5 TB.

## Performance Characteristics
- **First-byte latency:** ~10-50 ms typical for small objects.
- **Throughput:** essentially unlimited via parallelism — single-host throughput depends on instance NIC; aggregated GB/s easy with parallel transfers.
- **Multipart uploads** essential for objects > ~100 MB.
- **S3 Transfer Acceleration** uses CloudFront edge for upload speedup over long distances.
- **Bottlenecks:** single-prefix RPS limits, single-stream throughput, regional egress cost.

## Trade-offs

| Strength | Weakness |
|---|---|
| 11 nines durability, multi-AZ by default | Egress is expensive; cross-region transfer adds up |
| Effectively unlimited capacity | Per-prefix RPS limits at scale |
| Strongly consistent (since 2020) | No filesystem semantics (no append/rename) |
| Tiered classes (hot → archive) for cost | Glacier retrieval is slow + has minimum durations |
| Native integration with most AWS services | Lock-in to AWS API surface (mitigated by S3-compatible APIs elsewhere) |
| Object Lock for compliance / immutability | Lifecycle / versioning / replication add operational complexity |
| Mature ecosystem, every tool speaks S3 | Cost surprises from request count / lifecycle transitions |

## Common HLD Patterns
- **Static site hosting:** S3 (origin) + CloudFront (CDN) + Route 53 (DNS) + ACM (TLS).
- **Data lake substrate:** S3 (storage) + Glue Catalog (metadata) + Athena/EMR/Redshift Spectrum (compute).
- **Backup / archive:** Lifecycle rules tier objects from Standard → IA → Glacier Flexible → Deep Archive over time.
- **Cross-region DR:** CRR + Multi-Region Access Points for failover.
- **Pre-signed uploads:** clients upload directly to S3 via pre-signed URLs; backend never proxies bytes.
- **Event-driven pipelines:** S3 → SNS / EventBridge / SQS / Lambda for object-created notifications.

## Common Pitfalls / Gotchas
- **Public buckets** — long-running source of leaks; use Block Public Access at account + bucket level.
- **Overwrite vs version** — without versioning, an overwrite is unrecoverable; for important data, enable versioning + MFA-delete.
- **Lifecycle gotchas** — minimum storage duration (30 days for IA, 90/180 for Glacier); transitioning small files costs more than keeping them.
- **Eventual consistency in CloudFront** — invalidations take minutes; design URLs to include version/hash.
- **Listing cost / latency** — `ListObjectsV2` is paginated; for big buckets use S3 Inventory or maintain your own catalog (Iceberg/Hudi do this).
- **Cross-region egress charges** — design data placement to avoid surprise bills.
- **Per-prefix throttling** — when partitioning data, randomize prefix or bucket by hash for write-heavy workloads.
- **Strong consistency + caches** — your application cache (Redis, CDN) is still eventually consistent; invalidate appropriately.

## Interview Cheat Sheet
- **Tagline:** Globally durable HTTP-addressable object store; the foundation of AWS data infrastructure.
- **Best at:** lakes, backups, archives, static assets, ML datasets.
- **Worst at:** small-object random reads at extreme RPS on one prefix, filesystem semantics, OLTP.
- **Scale:** trillions of objects, exabyte buckets, millions RPS per region.
- **Shard by:** auto-prefix-sharding; user controls via key naming; multi-segment prefixes spread load.
- **Consistency:** strong read-after-write and list-after-write (since Dec 2020).
- **Replicates how:** multi-AZ within region by default; CRR / SRR async; MRAP for active-active routing.
- **Killer alternative:** Google Cloud Storage, Azure Blob, MinIO (self-hosted S3-compatible), Cloudflare R2 (no egress fees).
